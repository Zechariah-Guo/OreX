# Requirements Document

## Introduction

Two-Factor Authentication (2FA) adds TOTP-based verification as an optional security layer for OreX player accounts. Players can enable 2FA from their Settings page using any TOTP-compatible authenticator app (such as Google Authenticator). Once enabled, login requires both the password and a time-based 6-digit code. Single-use backup recovery codes provide account access if the authenticator device is lost.

## Glossary

- **Two_Factor_Authentication**: An optional security feature requiring a time-based one-time password in addition to the account password during login
- **TOTP**: Time-based One-Time Password — a 6-digit code generated every 30 seconds from a shared secret using the HMAC-based algorithm defined in RFC 6238
- **TOTP_Secret**: A base32-encoded cryptographic key stored on the player's account and shared with their authenticator app, used to generate valid TOTP codes
- **pyotp**: The Python library used for TOTP secret generation, provisioning URI creation, and code verification
- **qrcode**: The Python library used to generate QR code images from TOTP provisioning URIs
- **Authenticator_App**: A mobile application (such as Google Authenticator, Authy, or Microsoft Authenticator) that generates TOTP codes from a stored secret
- **QR_Code**: A scannable image generated by the qrcode library encoding the TOTP provisioning URI in the format `otpauth://totp/OreX:{username}?secret={secret}&issuer=OreX`
- **Manual_Key**: The base32-encoded TOTP_Secret displayed as text for players who cannot scan the QR_Code
- **Backup_Code**: A single-use alphanumeric recovery code that can substitute for a TOTP code during login verification
- **Two_Factor_Challenge_Page**: A dedicated page shown after successful password verification where the player enters their 6-digit TOTP code or a Backup_Code
- **Settings_Page**: The existing user settings interface where 2FA setup and management controls reside
- **Auth_System**: The Flask authentication subsystem handling login, registration, session management, and now 2FA verification
- **Rate_Limiter**: The existing mechanism restricting login attempts to 5 per 15-minute window per IP address
- **Pending_2FA_Session**: A server-side session state indicating the player has passed password verification but has not yet completed 2FA verification — the player is not considered authenticated in this state

## Requirements

### Requirement 1: 2FA Setup Initiation

**User Story:** As a player, I want to begin the 2FA setup process from my Settings page, so that I can add an extra layer of security to my account.

#### Acceptance Criteria

1. THE Settings_Page SHALL display a "Two-Factor Authentication" section within the Security area showing the current 2FA status (enabled or disabled) for the logged-in player
2. WHILE 2FA is disabled for a player, THE Settings_Page SHALL display an "Enable 2FA" button in the Two-Factor Authentication section
3. WHEN a player clicks the "Enable 2FA" button, THE Auth_System SHALL generate a new TOTP_Secret using the pyotp library and display the 2FA setup interface
4. WHEN the 2FA setup interface is displayed, THE Auth_System SHALL render a QR_Code (generated using the qrcode library) encoding the provisioning URI in the format `otpauth://totp/OreX:{username}?secret={secret}&issuer=OreX`
5. WHEN the 2FA setup interface is displayed, THE Auth_System SHALL render the Manual_Key as selectable text below the QR_Code for players who cannot scan

### Requirement 2: 2FA Setup Confirmation

**User Story:** As a player setting up 2FA, I want to verify that my authenticator app is correctly configured before activation, so that I do not lock myself out of my account.

#### Acceptance Criteria

1. WHEN the 2FA setup interface is displayed, THE Auth_System SHALL require the player to enter a valid 6-digit TOTP code from their Authenticator_App to confirm successful configuration
2. WHEN the player submits a valid TOTP code during setup, THE Auth_System SHALL persist the TOTP_Secret to the player's account and mark 2FA as enabled
3. IF the player submits an invalid TOTP code during setup, THEN THE Auth_System SHALL display an error message and allow the player to retry without regenerating the secret
4. THE Auth_System SHALL accept the current TOTP code, the immediately preceding code, and the immediately following code during setup verification to accommodate clock skew (a window of plus or minus one 30-second interval)

### Requirement 3: Backup Code Generation

**User Story:** As a player who has just enabled 2FA, I want to receive single-use backup recovery codes, so that I can regain access to my account if I lose my authenticator device.

#### Acceptance Criteria

1. WHEN a player successfully confirms 2FA setup, THE Auth_System SHALL generate exactly 8 unique single-use Backup_Codes for that player
2. WHEN Backup_Codes are generated, THE Auth_System SHALL display all 8 codes on-screen exactly once with a clear warning that the codes will not be shown again
3. THE Auth_System SHALL store Backup_Codes in the database in a hashed form (not plaintext) to protect against database compromise
4. THE Auth_System SHALL generate each Backup_Code as an 8-character alphanumeric string using a cryptographically secure random source

### Requirement 4: Login 2FA Challenge

**User Story:** As a player with 2FA enabled, I want to be prompted for my TOTP code on a separate page after entering my password, so that my account has two distinct verification steps.

#### Acceptance Criteria

1. WHEN a player with 2FA enabled submits valid credentials (username and password), THE Auth_System SHALL create a Pending_2FA_Session and redirect the player to the Two_Factor_Challenge_Page instead of completing the login
2. WHILE a Pending_2FA_Session is active, THE Auth_System SHALL NOT grant the player access to any authenticated routes or resources
3. THE Two_Factor_Challenge_Page SHALL present a single input field for the 6-digit TOTP code and a submission button
4. THE Two_Factor_Challenge_Page SHALL display a link or option to use a Backup_Code instead of a TOTP code
5. WHEN the player submits a valid TOTP code on the Two_Factor_Challenge_Page, THE Auth_System SHALL complete the login by fully authenticating the session and redirecting the player to the dashboard
6. THE Auth_System SHALL verify TOTP codes using the pyotp library and accept the current code, the immediately preceding code, and the immediately following code during login verification to accommodate clock skew

### Requirement 5: Backup Code Login

**User Story:** As a player who has lost access to their authenticator device, I want to use a backup recovery code to log in, so that I can regain access to my account.

#### Acceptance Criteria

1. WHEN a player submits a valid Backup_Code on the Two_Factor_Challenge_Page, THE Auth_System SHALL complete the login by fully authenticating the session
2. WHEN a Backup_Code is used successfully, THE Auth_System SHALL invalidate that specific code permanently so it cannot be reused
3. IF a player submits an invalid or already-used Backup_Code, THEN THE Auth_System SHALL reject the attempt and display an error message
4. WHEN the number of remaining valid Backup_Codes for a player reaches zero, THE Auth_System SHALL still require TOTP verification for login (manual recovery is out of scope)

### Requirement 6: 2FA Challenge Rate Limiting

**User Story:** As a system administrator, I want 2FA verification attempts to be rate limited, so that brute-force attacks against 6-digit codes are mitigated.

#### Acceptance Criteria

1. THE Auth_System SHALL apply the existing Rate_Limiter configuration (5 attempts per 15-minute window per IP address) to the Two_Factor_Challenge_Page submissions
2. WHEN an IP address exceeds the rate limit on the Two_Factor_Challenge_Page, THE Auth_System SHALL reject further verification attempts and display a retry message with the remaining lockout time in minutes
3. THE Auth_System SHALL count failed TOTP code submissions and failed Backup_Code submissions toward the same rate limit counter as failed password attempts

### Requirement 7: Pending Session Security

**User Story:** As a player, I want my pending 2FA session to expire automatically, so that an unattended browser cannot be used to bypass the second factor indefinitely.

#### Acceptance Criteria

1. WHEN a Pending_2FA_Session is created, THE Auth_System SHALL set an expiration time of 5 minutes from creation
2. IF a player attempts to submit a TOTP code or Backup_Code after the Pending_2FA_Session has expired, THEN THE Auth_System SHALL reject the attempt, invalidate the session, and redirect the player to the login page
3. WHILE a Pending_2FA_Session is active, THE Auth_System SHALL store only the player's user ID and the session creation timestamp — the Auth_System SHALL NOT store the password or TOTP_Secret in the session
4. IF a player navigates away from the Two_Factor_Challenge_Page without completing verification, THE Auth_System SHALL allow the player to return and complete verification within the 5-minute window using the same Pending_2FA_Session

### Requirement 8: Disable 2FA

**User Story:** As a player with 2FA enabled, I want to disable 2FA from my Settings page, so that I can simplify my login process if I choose.

#### Acceptance Criteria

1. WHILE 2FA is enabled for a player, THE Settings_Page SHALL display a "Disable 2FA" button in the Two-Factor Authentication section
2. WHEN a player clicks the "Disable 2FA" button, THE Auth_System SHALL require the player to enter a valid TOTP code or Backup_Code to confirm their identity before disabling
3. WHEN the player provides valid confirmation, THE Auth_System SHALL remove the TOTP_Secret and all remaining Backup_Codes from the player's account and mark 2FA as disabled
4. WHEN 2FA is disabled, THE Auth_System SHALL no longer present the Two_Factor_Challenge_Page during login for that player

### Requirement 9: TOTP Secret Storage Security

**User Story:** As a system administrator, I want TOTP secrets stored securely in the database, so that a database breach does not immediately compromise all 2FA protections.

#### Acceptance Criteria

1. THE Auth_System SHALL encrypt the TOTP_Secret using a server-side encryption key before storing it in the database
2. THE Auth_System SHALL derive the encryption key from the application SECRET_KEY using a key derivation function
3. THE Auth_System SHALL decrypt the TOTP_Secret only at the moment of TOTP code verification and discard the plaintext immediately after use
4. THE Auth_System SHALL store Backup_Codes using a one-way hash (the same hashing approach used for passwords) so that plaintext codes cannot be recovered from the database

### Requirement 10: Account Lifecycle Interactions

**User Story:** As a player, I want clarity on how 2FA interacts with account reset and deletion, so that there are no surprises about my security configuration.

#### Acceptance Criteria

1. WHEN a player resets their account, THE Auth_System SHALL retain all 2FA configuration (TOTP_Secret, Backup_Codes, and enabled status) without modification
2. WHEN a player with 2FA enabled resets their account, THE Auth_System SHALL continue to require TOTP verification on subsequent logins
3. WHEN a player deletes their account, THE Auth_System SHALL permanently remove all 2FA data (TOTP_Secret, Backup_Codes, and enabled status) along with all other account data
